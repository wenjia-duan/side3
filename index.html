ZiaZag Conversion [Medium]

题目要求

  • The string “PAYPALISHIRING” is written in a zigzag pattern on a given number of rows like this: (you may want to display this pattern in a fixed font for better legibility)

即：将输入字符串按列蛇形排列，最后按行打印结果字符串，两列之间的空格不打印

分析

算法分析

设行数为numRows, 输入字符串为s，输出字符串为d
算法并不复杂，从示例可以很容易分析出如下特征：

  1. 蛇形字符串为周期排列，且每个周期内的元素数量为2*numRows - 2。
  2. 每个周期内第0行和第N行的元素数量为1，中间行的元素数量可能为1，也可能为2。
  3. s与d的映射关系如下：
  • 第0行 d[i] = s[(2*numRows-2)*i] (i为列序号)
  • 第N行 d[i] = s[(2*numRows-2)*i +numRows-1] (i为列序号)
  • 中间行: 奇数列： d[i] = s[(numRows-1)*i + c], 偶数列： d[i] = s[(numRows-1)*i + 2*numRows - 2 - c] (c为行序号）

分析至此，算法也就呼之欲出了，有两种解法：

  1. 按列扫描
    即按照列的行进顺序依次计算结果，问题是输出的结果不是连续的，需要计算每行的偏移。

  2. 按行扫描
    即按行计算结果，无需计算每行的偏移，结果是按顺序输出的。

Review

标题

Branch and loop Reorganization to Prevent Mispredicts
重构分支和循环结构以避免分支预测失效

内容

背景

现代处理器为了提高指令的处理速度，大多采用多级的流水线来并行处理指令。这意味着后续指令不会等到前置指令执行完成后才执行。

这样造成的问题是：如果处理器碰到一个条件分支，但是条件指令还没执行完并输出结果，那处理器是如何知道应该使用哪个分支呢？
这就是需要引入分支预测(branch prediction)的原因。
分支预测是处理器用来决定使用哪个条件分支的算法，如果预测失败(mispredict)的话，将导致处理器清空预取的指令并从正确的指令重新开始，这通常会造成几十个CPU处理周期的性能损失。如果处理器的流水线深度很深的话，性能损失会更大。
这篇文章介绍了多种处理器使用的分支预测的算法并提供了一些如何避免分支预测失败的Tips。

条件分支的例子

如下是一些条件分支编译成汇编语言后的例子。

  • IFELSE

  • WHILE

  • DOWHILE

这里的分支跳转可以分为两种类型：

  • Forward Branch
    跳转的地址大于当前PC，比如上面IFELSE分支的NextIfElse1

  • Backward Branch
    跳转的地址小于当前PC，比如上面DOWHILE分支的DoWhileLoop

  • not taken
    not taken指的是非跳转的分支，即判定跳转条件为false，不执行跳转。
    例如IFELSE分支中的jne后的代码，即(data == 0)的分支

  • taken
    taken指的是跳转的分支，即判定跳转条件为true，执行跳转
    例如DoWHILE分支中的jg分支，即(data>0)的分支

分支预测的类型

存在两种类型的分支预测：

  • 静态分支预测:
    用于处理器第一次遇到条件分支时
  • 动态分支预测
    用于后续执行的条件分支

静态分支预测(static branch predic)

静态分支预测用于处理器还没有收集到分支数据的情况下，通常是第一次运行至该条件分支时。

动态分支预测(dynamic branch predict)

动态分支预测是通过记录处理器历史遇到的分支条件数据来实现的(history log)，历史记录数据包括每个分支的数据并记录以及分支是否被taken。这种分支历史记录(branch history log)被称为Branch Target Buffer(BTB)。每当处理器处理一个分支时，就会更新BTB相应的信息来反映其使用了哪个分支。
BTB记录了分支的地址以及分支执行的方向等简要历史信息。地址字段起的作用类似于BTB的索引，这条索引指向的记录指定了该分支是否应该被taken。在Pentinum4中，有16个bit位用来标记一个分支是否应该被taken。这些bit位就像一个环形缓冲区，每次检查BTB时就会去检查这些bit位的值。

如下是一个backward branch(即C++中的 do-while 循环）里BTB的例子，这个do-while有4个循环，且所有的BTB都已经被填上了对应的值。

$$ Full BTB Branch Entry (T for “taken”, N for “not taken”) $$

在这个例子中，do-while循环已经执行了很多次，每次执行都包含固定的4次循环。现在这些循环的执行历史已经存储在了BTB中，当这个循环重新执行的时候，就不会存在任何的分支预测错误和相应的性能损失。
但是值得注意的是，BTB的长度是有限的，一旦所有的BTB单元都被使用了，则较早写入的单元会被覆盖。
因为BTB在Pentinum 4处理器中只有16个单元的长度，所以如果循环的长度超过16，则分支预测会失效。

此为第一篇：初识paxos

初识paxos

什么是paxos

paxos是一种共识算法(consensus algorithm), 由Leslie Lamport在1990年提出。

共识即对某个提议(proposal)达成一致。为了达成共识，需要满足如下条件：

  • 只有一个提议的值可以被选中(chosen)
  • 只会选中一个值
  • 系统不会感知到一个值被选中除非大家达成了共识

即：共识确定的是一个值，且该值只有一个(常量).

为什么需要共识(paxos)?

paxos解决的是分布式系统的一致性问题。

试想在分布式系统中的N个节点,任意一个节点随时都可能宕机，节点彼此间只能通过“消息”来传递信息，但是此消息在传输过程中可能：

  • 丢失：消息没有到达对方
  • 乱序：消息到达的顺序与发送的顺序不一致
  • 但是不会被篡改

如何才能保证在上述条件下，分布式系统中的N个节点可以保持一致呢？这就是paxos算法要解决的问题。

或许你会说：这不是很简单吗，只要让分布式系统的所有节点都受到消息不就达成共识吗？

可是，在上面描述的条件下，只要有一个节点无法通信，系统就无法达成共识。

或许你还会说：那就不要用分布式系统了，只有一个节点就好了，节点间也不需要通信

可是，如果只有一个节点，系统的可用性是非常糟糕的。

或许你还会说：按照分布式系统的CAP理论，系统的一致性，可用性和分区容忍性是无法同时保证的。

嗯，paxos并没有推翻CAP理论，他只是在保证了一致性的前提下，最大化了系统的可用性(多数派)

追问：那quorum(NRW)协议也可以保证一致性呀

Paxos算法描述(基本上是翻译paxos simple这篇论文）

为了选中一个值，我们假设系统中存在一个提议者(proposer) 以及若干个接收者(acceptor)和若干个学习者(learner),只要足够多的接收者（大多数, >N/2)确认(accept)了提议者提议的值，我们就认为这个值被选中(chosen)。因为任意两个多数派必定有共同的接收者，因此被选中的值是唯一的。
另外，我们假定每个提议都有一个自增的编号，且不同的提议者提出的提议的编号不会重叠(这有很多种实现方法，比如分号段）
同时算法提出如下要求：

P1: 接收者(acceptor)必须接受他接收到的第一个提议(proposal)

An acceptor must accept the first proposal that it receives

P2: 如果一个值为v的提议(编号为n)被选中，那所有编号比n大且被选中的提议的值也是v

If a proposal with value v is chosen, then every higher-numbered proposal that is chosen has value v

P2a:如果一个值为v的提议(编号为n)被选中，那被接收者确认的所有编号比n大的提议的值为v

If a proposal with value v is chosen, then every higher-numbered proposal accepted by any acceptor has value v

P2b:如果一个值为v的提议(编号为n)被选中，那由提议者提议的所有编号比n大的提议的值为v

If a proposal with value v is chosen, then every higher-numbered proposal issued by any proposal has value v

P2c:

For any value v and n, if a proposal with value v and number n is issued, there is a set S consisting of a majority of
acceptors such that either (a) no acceptor in S has accepted any proposal numbered less than n, or (b) v is the value of the highest-numbered proprosal among all proposals numbered less than n accepted by acceptos in S.

由p2 -> p2a -> p2b 其实是很自然的，但是p2c就会比较晦涩些。

根据上述算法的描述， 我们可以把算法的执行过程分为三部分：

1. 提议者(proposal)

P1: 提议者提出一个提议P，序号为n，其值为v

P1a: 如果接收者接受了这个提议 $P_n$，则接受者承诺不会确认序号比n小的提议，同时，如果接受者已经确认(accept)了一个提议，则将已确认的提议中序号最大的提议的值一并返回

P2: 如果提议者确认多数派接受者已经接受了此提议，如果接收者未返回提议的值，则该提议Pn的值v被选中(chosen)，如果接收者返回了已确认的值，按照P2b的要求，则接受者返回的值被选中(chosen)

P2b: 按照P2中选中的值，向所有接收者发送确认消息(accept)
P2c: 如果接收者返回确认消息不可被确认，则该提议被终止，回到P1，重新发起提议。

2. 接受者(acceptor)

A1: 接受者收到提议者发出的提议，需要为n，其值为v

A1a: 如果接收者第一次接受到提议，则将该提议的序号n序列化存储，并告知提议者者已接受

A1b: 如果接收者已经接受到了其他提议，则承诺不会确认(accept)序号小于n的提议。如果接收者已经确认了(accept)某个提议，则将已确认的提议中的序号最大的提议的值返回给提议者

A2: 接受者收到提议者发出的确认(accept)消息

A2a: 如果确认消息的序号小于接受者存储的承诺的序号，则告知提议者该消息不可确认。

A2b: 如果确认消息的序号等于接收者存储的序号，则告知提议者该消息已确认。

3. 学习者(learner)

按下不表

个人知识管理   2019-04-26

今天看到一篇很好的关于个人知识管理的文章，简单总结了一下。

为什么要做知识管理

互联网工具的发展降低了分享和协作的成本，大量的知识被生产出来，造成信息过载，导致如下两个问题：

1. 知识的稀缺性

  • 信息的寿命正比于信息的价值
  • 互联网上大量无价值的信息稀释了真正有价值的知识的含金量
  • 真正有价值的知识可能只存在于“小圈子”中，并未在互联网上传播

2. 大脑处理能力与信息过载的冲突

  • 信息量飙升
  • 大脑处理能力无本质提升

问题：

如何找到真正有价值的信息？

如何提升信息处理的效率？

打法：

  • 聚焦核心
  • 做减法
  • 以慢为快，以少胜多

如何做个人知识管理

一个公式：B=PACK

个人知识管理的要素: 

B: 个人的品牌/标准
P: Product 产品系统
A: Action 行动系统
C: Cognize 认知系统
K: Knowlege 知识系统

时间，人，信息和产品是个人知识管理的四大底层要素。类似于人工智能中的算法、算力和数据。

- 行动系统类似于算力，解决怎样持续行动的问题
- 认知系统类似于算法，解决怎么进步的问题
- 知识系统类似于数据，未进步提供有效数据

两个步骤：知行合一

认知需要和行为相匹配，用查理.芒格的话来总结就是：“找到一个简单，基本的原则，非常严格地按照这个原则行事”
同时，这个原则需要与我们的认知相匹配。

不同类型人才的思考模式：
- 顶级人才靠"精神" - 领袖：人活着就是为了改变世界
- 一流人才靠身份 - 觉醒者：因为我是XXX，所以我会XXX
- 二流人才靠信念/价值观/规条 - 战略家：什么才是更重要的
- 三流人才靠能力 - 战术家：方法总比问题多
- 四流人才靠行为 - 行动派：我还不够努力
- 五流人才靠环境 - 怨妇：都是你们的错

4+N的倒T型知识结构

个人知识管理的结构是一个4+N的倒T型结构, 4即行为系统、认知系统、知识系统、产品系统构成了最基本的横线。

  • 通过刻意练习和习惯养成锻炼产品系统
  • 通过系统思维、沟通交流、理解人性提升认知系统
  • 通过用户体验、技术理解力、商业模式锤炼产品系统
  • 通过正确的分类与连接建立知识系统

在此基础上，可以建立不同的垂直业务线，将四大系统导入到不同的具体领域中去，即归纳与演绎的过程

五步法

  • 问： 问题的深度决定了行动的深度，做事之前先问why，再问what，再问how
  • 集： 获取知识的渠道因人而异，切忌贪多。 类型上选择一两个领域深耕，来源上舍弃过多渠道，选择权威的渠道。
  • 理： 知识管理不能只将知识收藏起来，隔三差五得翻看就可以，而是要对一个领域建立清晰细致的心理表征，需要注意知识点间的连接和分类。（还有一个熟练度的问题，对于核心知识点，需要做到即问即答，出口成章）
  • 享：知识管理的成果需要实践的检验。沾些文章，进行演讲，分享都是“享”的形式，“享”的目的是为了获取反馈，并且在准备“享”的过程中也能倒逼问、集、理的进行。
  • 思： 思在于找到差值，即“享”之前的预期与“享”之后的反馈，进而反思结果的正确性并指导下一个阶段的五步法实践，形成正向的循环，持续打磨知识管理。
