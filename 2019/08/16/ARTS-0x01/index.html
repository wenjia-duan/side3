Algorithm

ZiaZag Conversion [Medium]

题目要求

  • The string “PAYPALISHIRING” is written in a zigzag pattern on a given number of rows like this: (you may want to display this pattern in a fixed font for better legibility)

即：将输入字符串按列蛇形排列，最后按行打印结果字符串，两列之间的空格不打印

分析

算法分析

设行数为numRows, 输入字符串为s，输出字符串为d
算法并不复杂，从示例可以很容易分析出如下特征：

  1. 蛇形字符串为周期排列，且每个周期内的元素数量为2*numRows - 2。
  2. 每个周期内第0行和第N行的元素数量为1，中间行的元素数量可能为1，也可能为2。
  3. s与d的映射关系如下：
  • 第0行 d[i] = s[(2*numRows-2)*i] (i为列序号)
  • 第N行 d[i] = s[(2*numRows-2)*i +numRows-1] (i为列序号)
  • 中间行: 奇数列： d[i] = s[(numRows-1)*i + c], 偶数列： d[i] = s[(numRows-1)*i + 2*numRows - 2 - c] (c为行序号）

分析至此，算法也就呼之欲出了，有两种解法：

  1. 按列扫描
    即按照列的行进顺序依次计算结果，问题是输出的结果不是连续的，需要计算每行的偏移。

  2. 按行扫描
    即按行计算结果，无需计算每行的偏移，结果是按顺序输出的。

Review

标题

Branch and loop Reorganization to Prevent Mispredicts
重构分支和循环结构以避免分支预测失效

内容

背景

现代处理器为了提高指令的处理速度，大多采用多级的流水线来并行处理指令。这意味着后续指令不会等到前置指令执行完成后才执行。

这样造成的问题是：如果处理器碰到一个条件分支，但是条件指令还没执行完并输出结果，那处理器是如何知道应该使用哪个分支呢？
这就是需要引入分支预测(branch prediction)的原因。
分支预测是处理器用来决定使用哪个条件分支的算法，如果预测失败(mispredict)的话，将导致处理器清空预取的指令并从正确的指令重新开始，这通常会造成几十个CPU处理周期的性能损失。如果处理器的流水线深度很深的话，性能损失会更大。
这篇文章介绍了多种处理器使用的分支预测的算法并提供了一些如何避免分支预测失败的Tips。

条件分支的例子

如下是一些条件分支编译成汇编语言后的例子。

  • IFELSE

  • WHILE

  • DOWHILE

这里的分支跳转可以分为两种类型：

  • Forward Branch
    跳转的地址大于当前PC，比如上面IFELSE分支的NextIfElse1

  • Backward Branch
    跳转的地址小于当前PC，比如上面DOWHILE分支的DoWhileLoop

  • not taken
    not taken指的是非跳转的分支，即判定跳转条件为false，不执行跳转。
    例如IFELSE分支中的jne后的代码，即(data == 0)的分支

  • taken
    taken指的是跳转的分支，即判定跳转条件为true，执行跳转
    例如DoWHILE分支中的jg分支，即(data>0)的分支

分支预测的类型

存在两种类型的分支预测：

  • 静态分支预测:
    用于处理器第一次遇到条件分支时
  • 动态分支预测
    用于后续执行的条件分支

静态分支预测(static branch predic)

静态分支预测用于处理器还没有收集到分支数据的情况下，通常是第一次运行至该条件分支时。
静态分支预测的规则很简单：

  • Forward Branch 默认使用not taken
  • Backward Branch 默认使用taken

    同时，Pentium 4处理器还引入了一个新的指令来显式指定预测的分支：
  • 0X3E - 指定预测分支为taken
  • 0x2E - 指定预测分支为not taken

动态分支预测(dynamic branch predict)

动态分支预测是通过记录处理器历史遇到的分支条件数据来实现的(history log)，历史记录数据包括每个分支的数据并记录以及分支是否被taken。这种分支历史记录(branch history log)被称为Branch Target Buffer(BTB)。每当处理器处理一个分支时，就会更新BTB相应的信息来反映其使用了哪个分支。
BTB记录了分支的地址以及分支执行的方向等简要历史信息。地址字段起的作用类似于BTB的索引，这条索引指向的记录指定了该分支是否应该被taken。在Pentinum4中，有16个bit位用来标记一个分支是否应该被taken。这些bit位就像一个环形缓冲区，每次检查BTB时就会去检查这些bit位的值。

如下是一个backward branch(即C++中的 do-while 循环）里BTB的例子，这个do-while有4个循环，且所有的BTB都已经被填上了对应的值。

$$ Full BTB Branch Entry (T for “taken”, N for “not taken”) $$

在这个例子中，do-while循环已经执行了很多次，每次执行都包含固定的4次循环。现在这些循环的执行历史已经存储在了BTB中，当这个循环重新执行的时候，就不会存在任何的分支预测错误和相应的性能损失。
但是值得注意的是，BTB的长度是有限的，一旦所有的BTB单元都被使用了，则较早写入的单元会被覆盖。
因为BTB在Pentinum 4处理器中只有16个单元的长度，所以如果循环的长度超过16，则分支预测会失效。
